而是把最强 Agent 接进工程流程

+

可审计、可回退、可验证的 AI 开源贡献工作台
human judgment × agent runtime

大家好，今天我想用 10 页讲清楚 RV-Insights 的核心判断。它不是再造一个通用 Agent，也不是把 skills 堆得更高。

我们真正要做的是把 Claude Code、Codex 这类成熟 coding agent 的完整运行时，放进一个真实工程流程里。人类负责流程、边界和质量判断，AI 负责搜索、修改、验证和材料准备。

这页先记住一个公式：完整运行时，加上贡献 Pipeline，等于可审计、可回退、可验证的开源贡献工作台。

01 / diagnosis

Agent 已经会写代码，贡献仍然卡在流程

01Find什么值得做
02Plan边界和风险
03Patch代码和测试
04Review正确性和回归
05Prove验证和材料

真实开源贡献的瓶颈，不是“模型能不能动手”，而是“过程能不能被人类理解、审查和恢复”。

execution is solved; contribution flow is not

先看问题。现在很多 coding agent 已经能读代码、改文件、跑测试，所以“能不能动手”不是最核心的瓶颈。

真正困难的是这五步：先判断什么贡献值得做，再规划边界和风险，然后写 patch，接着审查正确性，最后拿出验证结果和提交材料。

如果这些过程只藏在长聊天里，维护者很难判断 AI 为什么这么改，失败在哪里，下一步该继续还是返工。RV-Insights 要解决的就是这个流程问题。

02 / why generic wrappers drift

通用 Agent 内核和 skills 堆叠，不是稳定性的答案

drift

内核追不上运行时

模型、CLI、工具协议持续变化，自研内核会变成长期适配负担。

semantic

skills 会漂移

skills 能增强上下文，但稳定性仍依赖模型理解和任务边界。

opaque

长任务不可审计

一次性跑完复杂贡献，判断、失败和返工原因会消失在输出里。

RV-Insights 的选择：不把稳定性押在薄封装上，而是押在完整运行时和工程流程上。

runtime > wrapper, workflow > prompt pile

这里是这套产品判断里最重要的一步：通用 Agent 内核和 skills 堆叠，并不天然稳定。

如果我们自研通用内核，就要不断追底层模型、CLI 和工具协议的变化。skills 当然有价值，但它更像增强层，不能承担全部稳定性。

还有一个问题是长任务不可审计。单 Agent 一次跑完以后，过程判断和返工原因经常消失。所以 RV-Insights 把稳定性放在完整运行时和工程流程上，而不是薄封装上。

03 / product bet

不做通用内核，
做运行时接入层和流程编排层

继承能力底层 agent 升级，已接入范围内优先受益。
降低适配不重复实现文件、终端、工具链和上下文管理。
保持控制平台负责工作区、权限、记录、产物和 gate。
connect mature agents, orchestrate engineering work

所以 RV-Insights 的产品赌注很明确：不做通用 Agent 内核，做运行时接入层和流程编排层。

运行时接入层负责把成熟 coding agent 接进来，尽量继承它们原生的文件操作、终端执行、工具链和上下文能力。

流程编排层负责工程边界：工作区、权限、记录、产物、checkpoint 和人工 gate。这样我们不会跟底层 agent 赛跑，而是把它放进更适合真实贡献的系统里。

04 / architecture

系统只有三层责任，
边界必须清楚

Agent Mode调用完整 coding agent 运行时
Pipeline Mode把贡献拆成阶段和人工 gate
Platform Layer管理工作区、权限、记录和产物
agent mode + pipeline mode + platform layer

把架构展开，其实只有三层责任。

Agent Mode 解决直接使用 coding agent 的场景，重点是调用完整运行时。Pipeline Mode 解决复杂贡献任务，把它拆成探索、规划、开发、审核、测试这些阶段。Platform Layer 解决长期工程协作需要的东西：工作区、权限、MCP、skills、记录、产物和 checkpoint。

这里最关键的是边界清楚。执行能力来自底层 agent，流程和控制来自 RV-Insights。

full runtime inheritance

Agent 模式可以用一句话解释：选择运行时，就继承它。

用户在当前项目里选择 Claude Code、Codex 或其他 coding agent。RV-Insights 不把它压缩成普通聊天接口，而是尽量保留原生运行时的能力，包括文件、终端、工具链、权限确认和上下文管理。

这样做的收益是上游升级可以被继承。同时，RV-Insights 负责工作区契约：cwd、MCP、skills、权限策略和本地记录都绑定到这个项目。

06 / pipeline mode

Pipeline 模式：把人类工程经验，变成五个可审批阶段

01

Explorer

找到真正值得做的贡献点。

02

Planner

明确边界、风险和验证路径。

03

Developer

修改代码、补测试、整理 diff。

04

Reviewer

审查正确性、回归和测试缺口。

05

Tester

执行验证，留下结果和阻塞项。

AI 做节点内工作；人类在阶段之间决定继续、重跑还是反馈返工。

human gates before expensive steps

Pipeline Mode 不是多 Agent 炫技，而是把工程师处理贡献的经验结构化。

Explorer 先判断贡献点，Planner 确认边界和风险，Developer 进入实现，Reviewer 做只读审查，Tester 负责最终验证。

这里的人机分工很重要：节点内部尽量交给 AI，阶段之间保留人工 gate。也就是说，在投入下一阶段成本之前，人类可以继续、重跑，或者把反馈带回去返工。

07 / definition of done

“完成”不是模型说了算，
而是闭环通过

Developer产出 diff、测试补充和变更摘要
Reviewer发现问题就反馈回开发节点
Tester验证通过后整理 patch / PR notes / artifacts
review feedback loop + test gate

这一页定义什么叫完成。不是模型说“完成了”，而是 Developer、Reviewer、Tester 这条闭环真正通过。

Developer 产出 diff、测试补充和变更摘要。Reviewer 只读审查，如果发现正确性、边界或回归问题，反馈回到 Developer 重新处理。Tester 最后执行验证，只有通过以后才整理 patch、PR notes 和 artifacts。

所以这里的 outcome 是提交材料准备，不是在未验证前自动 PR 提交。

08 / trust model

控制权不是口号，
而是状态、记录和可恢复性

Permission危险操作进入确认路径。
Gate继续、重跑、反馈是状态。
RecordsJSONL 可回放过程。
Artifacts报告、diff、测试落盘。
Checkpoint中断后恢复审批。
Local工作区默认在本机。
auditability + recoverability + human gates

RV-Insights 不追求“全自动”的幻觉，它追求的是让 AI 在清晰边界里做更多事。

控制权不是一句产品文案，而是具体状态。权限请求要进确认路径；Pipeline gate 要能继续、重跑和反馈；JSONL 让过程可回放；artifacts 让结果能迁移；checkpoint 让长任务中断后能恢复。

这些机制合起来，回答维护者最关心的问题：为什么这样改，失败在哪里，能不能证明可用，出了问题能不能退回可理解状态。

RV-Insights

最后收束成三句话。

第一，不造内核，接入运行时。RV-Insights 不重新实现成熟 coding agent 已经做得很好的能力。

第二，不赌 prompt，沉淀流程。我们把开源贡献拆成探索、规划、开发、审核、测试，让 AI 在节点内执行，让人类在阶段之间判断。

第三，不卖全自动，交付可审计。最终我们交付的是可审查、可恢复、可验证的贡献材料，而不是一句“AI 已完成”。谢谢大家，后面可以进入 demo 或 Q&A。
