# RV-Insights 教程

更新时间：2026年5月6日  
适用范围：当前仓库主线实现  
当前主入口：`Pipeline` / `Agent`

---

## 1. 这是什么

RV-Insights 是一个本地优先的 Electron 桌面应用，用来把大模型、工作区、Skills、MCP 和文件系统操作组织成一套可持续使用的 AI 工作流。

当前版本里，普通用户会直接接触到两个主模式：

- **Pipeline**：结构化流水线，适合多阶段任务，强调“执行一个节点 -> 人工审核 -> 再继续”
- **Agent**：通用自主 Agent，适合开放式探索、编码、调试、读写文件和长任务协作

旧的 `Chat` 能力目前只作为隐藏回退保留，不再是默认用户路径，因此本教程不再把它作为主要使用方式。

---

## 2. 先理解两个主模式

### Pipeline 模式

Pipeline 是当前项目的默认入口。它不是自由对话，而是固定的工作流：

`Explorer -> Planner -> Developer -> Reviewer -> Tester`

当前实现特点：

- 每个节点完成后都会暂停，等待你人工审核
- `Developer` 和 `Reviewer` 之间支持多轮往返
- 底层由 LangGraph 在主进程编排
- 节点执行统一走 Claude Agent SDK 兼容链路
- 会话、记录、checkpoint 都存到本地 JSON / JSONL 文件

适合场景：

- 你希望任务有明确阶段和可追踪记录
- 你想强制插入人工审核，避免 Agent 一路跑到底
- 你在做研发任务、方案拆解、代码改造或测试验证

### Agent 模式

Agent 模式是更自由的工作区型助手。

它适合：

- 代码阅读、重构、调试、批量修改
- 文档整理、脚本编写、命令执行
- 安装或使用 Skills / MCP
- 长时间探索一个问题

如果你不确定用哪个模式：

- 需要“分阶段 + 每步审批”时，用 **Pipeline**
- 需要“直接让助手持续完成工作”时，用 **Agent**

---

## 3. 第一次使用前要做什么

至少完成下面 3 件事：

1. **配置模型渠道**
2. **选择 Agent 默认渠道**
3. **选择默认工作区**

如果这 3 步没完成，Pipeline 和 Agent 都不能稳定工作。

---

## 4. 渠道配置

进入：

`设置 -> 渠道管理`

你需要新增至少一个可用于 Agent SDK 的渠道。当前代码里，`Pipeline` 和 `Agent` 共享这套 Agent 兼容渠道能力。

### 推荐渠道类型

优先使用这些 **Agent 兼容供应商**：

- `Anthropic`
- `DeepSeek`
- `Kimi API`
- `Kimi Coding`

原因：

- 当前 Pipeline 节点执行依赖 Claude Agent SDK 兼容链路
- 如果渠道不兼容，节点执行会直接失败

### 配置时需要填写

- 渠道名称
- 供应商类型
- Base URL
- API Key
- 可用模型

完成后请确认：

- 渠道可正常测试连接
- 至少有一个模型处于启用状态

---

## 5. Agent 设置也是 Pipeline 的基础设置

进入：

`设置 -> Agent 配置`

虽然名字叫 Agent 配置，但当前实现里，Pipeline 也会复用这里的默认工作区和默认渠道。

请重点确认：

- **默认渠道**：选中一个可运行 Agent 的渠道
- **默认模型**：选择你要长期使用的模型
- **默认工作区**：选择或创建一个你实际工作的目录上下文
- **权限模式**：根据你的习惯选择更保守还是更自动

如果你看到 Pipeline 侧栏右上角的齿轮，它现在会直接跳到这部分配置。

---

## 6. 工作区、Skills 和 MCP

### 工作区

工作区是 Agent / Pipeline 的上下文边界。

它决定：

- 默认 cwd
- 可见文件
- 可启用的 Skills
- 可启用的 MCP 服务器
- 会话之间共享的工作区资源

### Skills

Skills 是给模型的可复用工作方法。

你可以把它理解成：

- 一组针对特定任务的操作说明
- 一种渐进式提供上下文的方式
- 比“把所有规则都塞进系统提示词”更稳定的方案

典型用途：

- 代码审查
- 测试驱动开发
- 文档编写
- LangGraph / Agent SDK / 后端模式等专项开发

### MCP

MCP 用来连接外部能力或数据源。

比如：

- 搜索服务
- GitHub
- 文档系统
- 自定义本地工具

当前项目里，工作区会分别维护自己的 MCP 和 Skills 配置。

## 7. Pipeline 怎么用

### 创建会话

在左侧 `Pipeline` 侧栏点击：

`新建 Pipeline`

创建后会进入一个新的 Pipeline 会话。

### 启动一次流水线

在主区域输入一个任务，例如：

- “帮我分析这个仓库当前的 pipeline 架构缺口”
- “请实现一个新功能，并给出阶段性审查”
- “帮我定位一个 bug，并在每个阶段停下来等我确认”

然后点击：

`启动 Pipeline`

### 运行时你会看到什么

你会依次看到：

- 顶部状态
- 当前阶段高亮
- Records 区域中的运行记录
- 节点完成后的人工审核卡片

### 人工审核动作

每个需要审核的节点会停下来，通常可以做 3 类动作：

- **通过**
- **驳回并回退**
- **重跑当前节点**

这意味着你不是被动接收结果，而是在参与流程控制。

### Records 是干什么的

Records 是流水线的本地事件记录视图。

里面会逐步看到：

- 用户输入
- 节点切换
- 节点输出摘要
- 审核请求
- 审核响应
- 错误信息

如果状态显示在运行中，但 Records 没有变化，通常说明：

- 渠道配置有问题
- 当前节点执行失败
- 正在等待下一个事件同步

## 8. Agent 怎么用

切到左上角：

`Agent`

然后你可以：

- 新建 Agent 会话
- 选择工作区
- 直接下达编码、调试、读写文件、安装 skill 等任务

Agent 更像一个持续工作的助手，而不是固定阶段流程。

比较常见的用法：

- “阅读这个模块并解释它的状态流转”
- “帮我修复类型错误并跑验证”
- “在当前工作区安装一个 skill”
- “使用 MCP / Skills 完成调研或改造”

---

## 9. 本地存储与数据位置

RV-Insights 当前是本地优先设计。

你会在本机看到类似这些目录：

- `~/.rv-insights/`
- 开发模式下也可能使用 `~/.rv-insights-dev/`

当前实现中常见的数据包括：

- 渠道配置
- 应用设置
- Agent 会话
- Pipeline 会话
- JSONL 记录
- 工作区配置
- checkpoint

如果你在调试 Pipeline，通常最有用的是这些文件：

- `pipeline-sessions.json`
- `pipeline-sessions/{sessionId}.jsonl`
- `pipeline-checkpoints/{sessionId}/...`

## 10. 当前版本你应该知道的限制

本教程基于当前仓库状态，不是历史版本说明。

请特别注意：

- `Pipeline` 已经是默认主入口
- `Chat` 不是当前主要用户路径
- Pipeline UI 目前已可用，但仍在持续迭代
- 某些旧文档或旧截图里如果还写着 `Chat / Agent / OpenAI Reviewer`，以当前项目实现为准，不以旧文档为准

当前实现里更可靠的事实是：

- Pipeline 和 Agent 共用 Agent 兼容渠道能力
- Pipeline 的节点执行统一走 Claude Agent SDK 兼容链路
- 审核暂停与恢复是 Pipeline 自己的一等能力，不走普通对话交互

---

## 11. 推荐的上手顺序

如果你今天第一次用 RV-Insights，建议按这个顺序：

1. 先配置渠道
2. 再配置 Agent 默认模型和默认工作区
3. 进入 Pipeline，新建一个简单任务体验完整审批流程
4. 再切到 Agent，体验更自由的工作方式
5. 最后再回到工作区里配置 Skills 和 MCP

---

## 12. 可以直接尝试的任务

### 适合 Pipeline

- “帮我分析这个仓库的架构问题，并每步停下来等我确认”
- “请把一个功能拆成实现计划、开发、审查、测试”
- “帮我修一个 bug，并在 reviewer 阶段给出明确是否通过”

### 适合 Agent

- “阅读这个目录，告诉我最关键的模块关系”
- “修复当前 typecheck 错误并解释原因”
- “帮我在当前工作区安装一个 skill”
- “把这段需求改成一份实现计划或一组测试”

---

## 13. 最后

如果你在使用中遇到问题，优先排查这几件事：

- 渠道是否真的可用
- 选中的模型是否启用
- 当前工作区是否正确
- 你是否把适合 Agent 的任务放进了 Pipeline，或者反过来

本教程的目标不是覆盖所有细节，而是让你基于当前项目状态快速建立正确心智模型：

- **Pipeline** 负责结构化、多阶段、可审批的任务
- **Agent** 负责开放式、自主执行的任务
- **工作区 / Skills / MCP** 负责把它们变成长期可复用的系统
